# agent-os v0.2: My Agent Now Knows Every Project I've Ever Worked On

Five weeks ago I released [agent-os](https://github.com/desertcache/agent-os), a persistence layer for Claude Code. Hooks, skills, and a self-correcting memory system that give your AI agent session continuity. The response was encouraging. The daily use was more so.

In those five weeks I used it every day across 20+ projects. The system worked. It also showed me exactly where it was weak. v0.2 fixes those weaknesses.

## The Problem With v0.1

v0.1 required you to manually register every project in a config file. Name, codebase path, vault folder. Fine when you have three projects. Tedious when you have twenty. Worse: when you create a new project and forget to register it, the agent doesn't know it exists. The very problem agent-os was built to solve -- context loss -- was baked into the setup.

The session-start skill was also flat. One project briefing, no index. If I opened Claude Code from my home directory (which I do constantly), it had no idea what was available. "No project detected. What are we working on?" Every time. I'd have to remember my own project names and type one in.

## What Changed

### Vault Auto-Discovery

The preloader now scans your vault folders at session start. Every folder under `Projects/` (or whatever directories you configure) gets indexed automatically. It reads each README, parses frontmatter for status and codebase path, extracts a summary from the body, and builds a full project index. No registration required.

Create a folder in your vault with a README, and the next time you start a session, the agent knows about it. Remove the folder, and it's gone from the index. The vault is the source of truth, not a config file.

The config file still exists as a fallback. If your project doesn't have a vault folder yet, you can register it in `agent-os.config.json` and the preloader will pick it up. But for most projects, just having a vault README with a `codebase:` frontmatter field is enough.

### Two-Phase Session Start

`/session-start` now has two phases.

**Phase 1** shows everything. If you're in your home directory, you get a sorted index of every project the agent knows about:

```
Session Context Loaded

## Projects
| Project     | Last Worked | Summary                        |
|-------------|------------|--------------------------------|
| my-api      | 3/31       | JWT auth, middleware refactor   |
| dashboard   | 3/30       | React 19, real-time charts     |
| agent-os    | 3/30       | Open-source Claude Code infra  |
| side-project| --         | Planned but not started        |

## Dormant
old-prototype (complete), abandoned-idea (deprecated)

What are we working on?
```

You pick a project. The agent loads deep context.

**Phase 2** reads the vault README, finds related session notes, checks for error patterns relevant to the project's tech stack, and presents a focused briefing:

```
Deep Context: my-api

Status: active
Codebase: ~/code/my-api
Last Session: 3/31 -- auth-middleware-fix

Current State:
- JWT refresh endpoint working
- Rate limiting added to auth routes
- Integration tests passing

Watch out for: Express 5 wildcard routes need /{*path} syntax

What's the focus?
```

If you `cd` into a project directory before opening Claude Code, Phase 1 is skipped entirely. The agent detects the project from your working directory and goes straight to the deep briefing. Zero friction.

### Session-Project Linking

This one was subtle but important. In v0.1, session handoff notes lived in a flat `Sessions/` folder. The agent had to guess which project a session belonged to by pattern-matching the filename. "2026-03-15-auth-refactor.md" -- is that the API project or the dashboard?

Now every session note includes a `project:` frontmatter field that matches the vault folder name exactly:

```yaml
---
date: 2026-03-31
tags: [session, auth, middleware]
project: my-api
---
```

The preloader does a two-pass mapping: first by frontmatter tag (reliable), then by filename slug (fallback for old notes). The result is that `/session-start` shows accurate "Last Worked" dates for every project, and deep context loads always find the right session history.

### New Skill: /red-team

I kept finding myself asking Claude to poke holes in plans before executing them. The pattern was consistent enough to formalize.

`/red-team` takes any plan, architecture, proposal, or decision and runs it through six adversarial lenses: assumptions, failure modes, edge cases, complexity, security, and human factors. Findings are ranked by severity (Critical / High / Medium / Low) with specific fixes, not vague concerns.

The key constraint: it has to be constructive. "This might fail" is useless. "X will fail when Y because Z -- fix by doing W" is useful. The skill enforces this in its output format.

### New Skill: /meeting-notes

After a meeting, dump the raw transcript or your rough notes and run `/meeting-notes`. It extracts structured output: summary, key discussions, decisions table, prioritized action items, follow-up emails, and unanswered questions. Saves to the vault with project linking.

Simple, but it saves 10 minutes of post-meeting formatting every time.

## The Architecture Shift

The big conceptual change in v0.2 is where the source of truth lives.

**v0.1**: `agent-os.config.json` is the source of truth. The preloader reads it, matches against it, trusts it.

**v0.2**: The vault is the source of truth. The preloader scans it, extracts from it, builds an index dynamically. The config file is a fallback and a place for machine-specific settings (formatter paths, CLI locations).

This matters because the vault is something you naturally maintain. When you update a project's status in its README, the index reflects it next session. When you archive a project by moving its folder, it disappears from the index. The system stays current because you're already doing the work that keeps it current.

## Try It

If you're already on v0.1, pull and re-run setup:

```bash
cd agent-os
git pull
node setup.js
```

Setup is idempotent -- it won't overwrite your existing memory files or vault content.

Fresh install:

```bash
git clone https://github.com/desertcache/agent-os.git
cd agent-os
node setup.js
```
